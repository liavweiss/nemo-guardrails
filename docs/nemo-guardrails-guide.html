What is NeMo Guardrails?

The NVIDIA NeMo Guardrails library is an open-source Python package for adding programmable guardrails to LLM-based applications. Guardrails make your LLM-based application safer and more secure by blocking inappropriate, off-topic, or malicious user inputs and LLM responses.

NVIDIA NeMo is a suite of microservices, tools, and libraries for building, deploying, and scaling LLM-based applications. NeMo Guardrails provides tools to build guardrails and integrate them at both development time (Python library) and production time (API server, Docker, or NeMo microservice). For full integration options, see the official Integrate Guardrails into Your Application guide.

What Problems Does It Solve?

🚫

Jailbreak Protection

Blocks malicious prompts trying to manipulate your AI into ignoring safety rules

🎯

Topic Control

Prevents AI from discussing forbidden or inappropriate topics

☣️

Content Safety

Filters out toxic, harmful, or inappropriate content in both inputs and outputs

🔐

PII Detection

Detects and masks Personally Identifiable Information to protect user privacy

✅

Response Validation

Validates AI responses for accuracy, safety, and compliance

🔧

Tool Access Control

Controls what external tools and APIs the AI can access and execute

How It Works

NeMo Guardrails uses an event-driven architecture that processes your LLM interactions through multiple checkpoints, ensuring safety at every stage.

Complete Request Flow

1. User Input - Raw message from the user
↓
2. Input Rails - Safety checks on user message (jailbreak, harmful content, PII)
↓
3. Retrieval Rails - Filter documents if using RAG (optional)
↓
4. Dialog Rails - Control conversation flow (optional)
↓
5. LLM Processing - Your main AI model generates response
↓
6. Execution Rails - Validate tool calls if AI wants to use external APIs (optional)
↓
7. Output Rails - Final safety checks on AI response
↓
8. Final Response - Safe, validated response to user
💡 Defense in Depth: Multiple checkpoints mean that even if something passes one stage, it can still be caught at another. This layered approach provides comprehensive protection.

The 5 Types of Guardrails (Rails)

NeMo Guardrails provides five types of "rails" (checkpoints) that operate at different stages of the LLM interaction pipeline. Each rail type serves a specific security purpose:

Stage 1: BEFORE LLM

🛡️ Input Rails

What it does:

Validates and sanitizes user inputs BEFORE they reach your LLM

Use Cases:

  • Block harmful, toxic, or inappropriate inputs
  • Catch jailbreak attempts ("Ignore previous instructions...")
  • Ensure users stay on allowed topics
  • Detect and mask sensitive data (SSN, credit cards)

Example:

User: "How to hack into a system?"
   ↓
Input Rail: BLOCKED! (Harmful content)
   ↓
Response: "I cannot help with that request."
Stage 2: RAG PIPELINE

📚 Retrieval Rails

What it does:

Filters and validates retrieved knowledge during RAG (Retrieval-Augmented Generation)

Use Cases:

  • Only use trusted and approved document sources
  • Ensure retrieved text chunks are relevant and safe
  • Filter out off-topic or outdated documents
  • Prevent sensitive data from retrieved docs reaching the LLM

Example:

User: "What's our company policy on X?"
   ↓
Retrieve documents from knowledge base
   ↓
Retrieval Rail: Filter outdated policies
   ↓
LLM receives only current, validated docs
Stage 3: CONVERSATION

🗣️ Dialog Rails

What it does:

Controls conversation flow and enforces policies across multiple messages

Use Cases:

  • Guide conversations along predefined paths
  • Keep users on track with business logic
  • Monitor ongoing conversation for sensitive data leaks
  • Enforce conversation state and boundaries

Example:

Turn 1: User asks about product A
Turn 2: User asks about pricing
Turn 3: User asks about competitor
   ↓
Dialog Rail: "Let's focus on our products"
Stage 4: TOOL CALLS

🔧 Execution Rails

What it does:

Validates and controls external actions the LLM wants to perform

Use Cases:

  • Check parameters before calling external tools
  • Verify results from external systems
  • Restrict which APIs the LLM can call
  • Prevent dangerous tool usage (delete, modify sensitive data)

Example:

LLM wants: delete_file("important_data.sql")
   ↓
Execution Rail: BLOCKED! (Dangerous action)
   ↓
Response: "Cannot perform destructive operations"
Stage 5: AFTER LLM

✅ Output Rails

What it does:

Evaluates and post-processes model responses BEFORE showing them to users

Use Cases:

  • Block unsafe or off-policy AI responses
  • Strip out accidentally generated PII or secrets
  • Verify accuracy against trusted sources
  • Ensure appropriate tone and language

Example:

LLM generates: "Your SSN is 123-45-6789..."
   ↓
Output Rail: BLOCKED! (PII detected)
   ↓
Response: "I cannot share personal information."
💡 Most Common: Input and Output Rails are the most frequently used because they form the first and last lines of defense. However, you can combine multiple rail types for comprehensive protection.

Requirements & Installation

System Requirements

🐍 Python Version

Python 3.10, 3.11, 3.12, or 3.13

💾 Memory

Minimum 4GB RAM (CPU only, no GPU required)

🔨 Compiler

C++ compiler (for Annoy library installation)

✅ NO NVIDIA Hardware Required!
NeMo Guardrails works with ANY LLM provider - OpenAI, Azure OpenAI, Anthropic (Claude), Google Vertex AI, Cohere, AWS Bedrock, vLLM, HuggingFace, or any self-hosted model. NVIDIA models are completely optional.

Installation

Basic Installation:

# Install core library
pip install nemoguardrails

# Or with specific provider support
pip install "nemoguardrails[openai]"      # OpenAI
pip install "nemoguardrails[nvidia]"      # NVIDIA models (optional)
pip install "nemoguardrails[all]"         # All integrations

Optional Features:

pip install "nemoguardrails[sdd]"         # Sensitive data detection
pip install "nemoguardrails[eval]"        # Evaluation tools
pip install "nemoguardrails[tracing]"     # OpenTelemetry tracing
pip install "nemoguardrails[jailbreak]"   # Jailbreak detection

How to Configure Guardrails

For the full schema (models, rails, prompts, all options) and a complete config example, see the Configuration YAML Schema Reference.

NeMo offers three configuration approaches:

✅ Approach 1: Built-In Self-Check

Recommended for most users

Uses LLM to check inputs/outputs against natural language policies

Files needed: config.yml + prompts.yml

No code required!

⚙️ Approach 2: Custom Colang Flows

For advanced control

Define custom logic and conversation flows (we use Colang 1.0)

Files needed: config.yml + config.co (+ optional actions.py)

When: Guard-only, keyword/pattern checks, custom business logic

🔄 Approach 3: Hybrid

Best of both worlds

Combine built-in checks with custom flows

Files needed: config.yml + prompts.yml + config.co (or rails.co)

When: Need both flexibility and simplicity

Approach 1: Built-In Self-Check (Recommended) 🌟

✅ Use This When:
  • You want content safety (block harmful, explicit, abusive content)
  • You want LLM-powered, context-aware checking
  • You don't need custom conversation flows
  • You want to get started quickly (5 minutes)
  • This is what we implemented successfully!

Step 1: config/config.yml

# Minimal configuration - activate built-in self-check rails
models:
  - type: main
    engine: google_genai
    model: gemini-2.5-flash
    parameters:
      google_api_key: ${GOOGLE_API_KEY}

# Activate the built-in self-check flows
rails:
  input:
    flows:
      - self check input    # LLM checks user input
  
  output:
    flows:
      - self check output   # LLM checks bot output

Step 2: config/prompts.yml (The Policies)

This file tells the LLM what to check for. It uses natural language policies:

prompts:
  # Define what user inputs are NOT allowed
  - task: self_check_input
    content: |-
      Your task is to check if the user message complies with policy.

      Company policy for user messages:
      - should not contain harmful data
      - should not ask to impersonate someone
      - should not contain explicit content
      - should not use abusive language
      - should not share sensitive information
      - should not contain code execution requests
      
      User message: "{{ user_input }}"
      
      Question: Should the user message be blocked (Yes or No)?
      Answer:

  # Define what bot outputs are NOT allowed
  - task: self_check_output
    content: |-
      Your task is to check if the bot message complies with policy.

      Company policy for bot:
      - messages should not contain explicit content
      - messages should not contain abusive language
      - messages should not contain harmful content
      - if a message is a refusal, should be polite

      Bot message: "{{ bot_response }}"

      Question: Should the message be blocked (Yes or No)?
      Answer:
💡 How It Works:
  1. User sends: "How to make a bomb?"
  2. NeMo calls LLM with self_check_input prompt → LLM says "Yes" (block it)
  3. NeMo returns: "I'm sorry, I can't respond to that." ✅ Blocked!

Makes 3 LLM calls per request: input check → main task → output check

⚠️ Important: This approach uses the LLM itself as the judge. It's powerful and context-aware but:
  • Uses more API calls (3x)
  • Slightly slower (extra ~300-500ms)
  • Better accuracy than keyword matching

✅ That's It! No Colang, No Python Code Needed

Just two YAML files and you have working, production-ready guardrails.

Approach 2: Custom Colang Flows ⚙️

🤔 What is Colang?

Colang (Conversational Language) is NeMo's Domain-Specific Language (DSL) for defining conversation flows and custom guardrail logic.

Why we use Colang 1.0 instead of Colang 2.x

We use Colang 1.0 for guard-only (no LLM) configurations. Colang 2.x is newer and still in beta; NVIDIA’s FAQ states it is not recommended for production and may cause instability. In practice, with Colang 2.x and the HTTP server in guard-only mode (dialog: false, no main LLM), the API often returns empty assistant content even when input rails run and emit block/allow messages—the response pipeline does not collect bot utterances correctly in that setup. Colang 1.0 is better tested for guard-only over /v1/chat/completions and returns the expected block/allow messages. The examples below use Colang 1.0.

✅ Use Colang When:
  • You need custom conversation flows (multi-turn dialog management)
  • You need keyword-based checks (faster than LLM checks)
  • You need conditional logic ("if user asks about X, then do Y")
  • You need to call custom Python functions
  • You want deterministic behavior (not LLM-based)

Example: Guard-only with Colang 1.0 (input rails, no LLM)

This example matches our project: Colang 1.0, no models section, and input rails that call Python actions. The rails.input.flows list runs in order; the last flow sends the allow response when nothing was blocked.

Step 1: config/config.yml
# Guards only, no LLM — Colang 1.x
colang_version: "1.0"

# No "models" section. Input rails run in order.
rails:
  input:
    flows:
      - check input rail
      - allow input
Step 2: config/config.co (Colang 1.0 flows)
# Colang 1.x — input rails only (guard-only, no LLM).

define bot blocked_harmful
  "I cannot help with that request as it may be harmful or illegal."

define bot blocked_sensitive
  "Please do not share passwords, API keys, or other sensitive data in chat."

define bot allow
  "Request allowed."

define subflow check input rail
  run $harmful = check_harmful_keywords(text=$user_message)
  if $harmful
    bot blocked_harmful
    stop
  run $sensitive = check_sensitive_patterns(text=$user_message)
  if $sensitive
    bot blocked_sensitive
    stop

define subflow allow input
  bot allow
  stop

✅ Colang Pros

  • ⚡ Faster (no extra LLM calls)
  • 💰 Cheaper (no check costs)
  • 🎯 Deterministic (predictable)
  • 🔧 Fine-grained control

❌ Colang Cons

  • 🔤 Keyword-based (easy to bypass)
  • 🛠️ Requires maintenance
  • 📚 Learning curve (new DSL)
  • 🤖 Less context-aware

Python actions (used by the example above)

Colang 1.0 calls Python with run $var = action_name(param=$value). The actions used by check input rail are:

config/actions.py
from nemoguardrails.actions import action

@action()
def check_harmful_keywords(text: str) -> bool:
    """Return True if the message should be blocked (harmful). No LLM."""
    # e.g. block phrases like "bomb", "hack", "kill"
    ...

@action()
def check_sensitive_patterns(text: str) -> bool:
    """Return True if the message looks like sharing secrets/PII. No LLM."""
    # e.g. "my password is ...", SSN, API keys
    ...
⚠️ Complexity Warning: Colang is powerful but adds complexity. Start with built-in self-check (Approach 1) unless you specifically need:
  • Custom dialog management
  • Integration with external systems
  • Complex conditional logic

Approach 3: Hybrid (Built-In + Custom) 🔄

✅ Use Hybrid When:
  • You want LLM-powered content safety (broad coverage)
  • Plus fast keyword checks for specific known threats
  • Plus custom business logic
  • Example: Block known attack patterns instantly (keywords), then LLM checks unknown patterns

Example: Layered Defense

config/config.yml
models:
  - type: main
    engine: google_genai
    model: gemini-2.5-flash
    parameters:
      google_api_key: ${GOOGLE_API_KEY}

rails:
  input:
    flows:
      - check harmful keywords      # Fast keyword check (Colang)
      - self check input           # Deep LLM check (prompts.yml)
  
  output:
    flows:
      - self check output          # LLM output validation
config/config.co (Colang 1.0 — quick keyword filter)
define bot refuse harmful
  "I cannot help with harmful requests."

define subflow check harmful keywords
  run $harmful = check_harmful_keywords(text=$user_message)
  if $harmful
    bot refuse harmful
    stop
config/prompts.yml (Deep LLM Check)
prompts:
  - task: self_check_input
    content: |-
      Check if the user message complies with safety policy.
      [... policies ...]
      
  - task: self_check_output
    content: |-
      Check if the bot message complies with safety policy.
      [... policies ...]

Hybrid Flow Diagram

User Input: "How to make a b0mb?" (bypasses keywords)
↓
Layer 1: check harmful keywords → ❌ No match (b0mb != bomb)
↓
Layer 2: self check input (LLM) → ✅ BLOCKED (understands intent)
↓
Return: "I'm sorry, I can't respond to that."

Benefits of Hybrid

  • ⚡ Fast: Known threats blocked instantly by keywords
  • 🛡️ Comprehensive: LLM catches novel/obfuscated threats
  • 🎯 Flexible: Custom logic for business rules
  • 💰 Cost-effective: Keywords reduce LLM calls for common cases

Using NeMo in a Kubernetes System

For our project we use NeMo Guardrails as a service in Kubernetes: run the Guardrails API Server in a pod and call it via REST. This matches the official approach of deploying guardrails as a separate component that other services call.

How it works: You run nemoguardrails server --config /path/to/config inside a container. The server exposes POST /v1/chat/completions (and GET /v1/rails/configs). Your app or another pod (e.g. an inference gateway) sends HTTP requests to the NeMo service; NeMo runs the configured rails and returns the response. No GPU required; the server runs on CPU.

Deploying in Kubernetes:

  • Image: Build a Docker image that installs nemoguardrails, copies your config directory (e.g. config/ with config.yml, optional rails.co, prompts.yml, actions.py), and runs nemoguardrails server --config /config --port 8000 --host 0.0.0.0. See Deploy with Docker.
  • Pod/Deployment: Run the image as a Deployment; mount a ConfigMap or embed config in the image. Expose port 8000.
  • Service: Create a ClusterIP (or NodePort/LoadBalancer) Service so other pods can call http://nemo-guardrails:8000.

Calling from another service: Send POST /v1/chat/completions with JSON body: config_id, messages, and optionally options (e.g. options.rails.dialog = false for guard-only). Response includes messages (assistant reply or refusal). See API Server Endpoints and Run the NeMo Guardrails Server.

Guardrails With Inference vs Without Inference

NeMo can be used in two main ways. Both use the same POST /v1/chat/completions endpoint; the difference is config and request options. Aligned with API Server Endpoints and Configuration YAML Schema Reference.

🟣

With Inference

Full pipeline: input rails → LLM generates reply → output rails. NeMo returns the assistant message. Default when you don’t set options.rails.dialog = false.

How to use

  • Config: models with type: main and your LLM; rails.input.flows / rails.output.flows as needed.
  • Request: POST /v1/chat/completions with config_id, messages; omit options.rails.dialog or leave dialog enabled.

✓ Pros

  • One service for guarding and generation
  • Consistent policy before and after the LLM

✗ Cons

  • Requires LLM and API key; latency and cost
🟢

Without Inference

Only rails run: dialog step skipped, no LLM reply. You get block/allow and an optional refusal message. Use when inference is elsewhere and NeMo is purely a guard.

How to use

  • Request: POST /v1/chat/completions with options.rails.dialog = false, input: true (and optionally output: true to validate a candidate reply).
  • Config: No models needed if rails are Colang-only or use non-LLM Python actions; no API key.

✓ Pros

  • No inference in NeMo; can run with no model/API key
  • Fast; good for guard-only in front of your own LLM

✗ Cons

  • Non-LLM rails (e.g. keywords) easier to bypass

Example: Non-Inference Usage (Guards Only)

Below is the pattern we use: Colang 1.0, no models, input rails only, and Python actions (harmful keywords + sensitive-data patterns). No LLM and no API key.

1. config.yml

Set colang_version: "1.0" and list input flow names. Flows run in order; the last one (allow input) sends the allow response when nothing was blocked.

colang_version: "1.0"

# No "models" section.
rails:
  input:
    flows:
      - check input rail
      - allow input

2. config.co (Colang 1.0) — bot messages and two input subflows:

define bot blocked_harmful
  "I cannot help with that request as it may be harmful or illegal."

define bot blocked_sensitive
  "Please do not share passwords, API keys, or other sensitive data in chat."

define bot allow
  "Request allowed."

define subflow check input rail
  run $harmful = check_harmful_keywords(text=$user_message)
  if $harmful
    bot blocked_harmful
    stop
  run $sensitive = check_sensitive_patterns(text=$user_message)
  if $sensitive
    bot blocked_sensitive
    stop

define subflow allow input
  bot allow
  stop

3. actions.py — two checks (no LLM):

from nemoguardrails.actions import action

@action()
def check_harmful_keywords(text: str) -> bool:
    """Return True if the message should be blocked (harmful). No LLM."""
    ...

@action()
def check_sensitive_patterns(text: str) -> bool:
    """Return True if the message looks like sharing secrets/PII. No LLM."""
    ...

4. Request — set options.rails.dialog = false so no reply is generated:

POST /v1/chat/completions
Content-Type: application/json

{
  "config_id": "default",
  "messages": [{"role": "user", "content": "How to make a bomb?"}],
  "options": {
    "rails": {
      "input": true,
      "output": false,
      "dialog": false
    }
  }
}

5. Response behavior

  • Blocked: check input rail runs; if harmful or sensitive, it does bot blocked_harmful or bot blocked_sensitive and stop. The API returns that message as the assistant content (e.g. "I cannot help with that request…" or "Please do not share passwords…").
  • Allowed: Neither check blocks, so the next flow allow input runs and does bot allow. The API returns "Request allowed." as the assistant content. Your client can then proceed (e.g. call your own LLM).

Summary: With Inference vs Without Inference

With inference: NeMo runs input rails, then generates the assistant reply using the configured main model, then runs output rails. You get a full chat completion from a single call. Use this when you want one service to both guard and generate, and are fine with providing an LLM and API key.

Without inference: NeMo runs only the rails you enable (input and/or output); the dialog step is skipped, so no reply is generated. You get a block/allow outcome and an optional refusal message. Use this when inference runs elsewhere (e.g. your own LLM or gateway) and you want NeMo purely as a guard—optionally with no model and no API key if your rails are Colang-only or use non-LLM Python actions.

What You Can Do Without Inference

  • Input validation: Run input rails only (options.rails.input = true, dialog = false). Block or allow the user message; no LLM reply.
  • Output validation: Validate a candidate reply from your own LLM: send the user message and the candidate assistant message in messages, set options.rails.output = true and dialog = false. NeMo runs output rails on that candidate (e.g. self-check or Colang flow) and returns block/allow.
  • Colang flows: Use Colang 1.0 flows in config.co (e.g. define subflow, run $var = action(...), bot, stop).
  • Python actions: Call @action() functions from Colang (e.g. keyword lists, regex, external APIs). No model needed if the action does not call an LLM.
  • No API key: When no rail uses an LLM, omit models in config and run with no API key.
  • Refusal message: When a rail blocks, it can set the assistant message (e.g. bot say "..." in Colang); that message is returned in the API response so the client can show it to the user.
